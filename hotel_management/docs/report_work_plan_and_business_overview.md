# 酒店管理系统课程设计分工与业务说明参考稿

## 1. 文档用途

本文档用于小组撰写数据库课程设计报告时统一口径。项目以 `T224源码` 酒店管理系统为实现载体，围绕数据库结构、数据流图、E-R 图、逻辑模型、物理模型、视图、存储过程、触发器和简单数据库交互演示等内容展开整理。

`hotel_management_answer.sql`

- 基础表与扩展表合计：24 张表
- 复杂视图：8 个
- 存储过程：8 个
- 触发器：8 个

## 3. 系统需求分析

### 3.1 系统综合需求

随着酒店业务信息化程度不断提高，传统依靠人工登记、纸质台账和电话沟通完成预订、入住、退房和结算的工作方式，已经难以满足现代酒店在效率、准确性和服务质量方面的要求。酒店管理系统需要围绕客户订房、员工接待、房态维护、账单结算、评价统计和后台管理等核心场景，对相关业务数据进行统一存储、统一处理和统一查询，从而保证酒店业务流程连续、信息传递及时、经营数据清晰可追踪。

本系统面向客户、员工和管理员三类主要角色展开设计。客户侧强调信息浏览、在线预订、账单查询和评价反馈；员工侧强调预订处理、入住登记、退房结算和房态维护；管理员侧强调基础数据维护、权限管理、经营统计和系统配置。结合系统当前表结构、业务流程和数据库对象设计，综合需求可概括如下：

1. 客户预订需求  
   系统应支持客户按入住日期、离店日期、房型、价格区间等条件检索可订客房，并能够根据客房价格、预订数量和价格策略自动计算预订金额。客户提交预订后，系统应生成唯一预订编号，保存客户信息、客房信息、预订日期、支付状态和订单状态，同时完成相应房间资源的占用控制，避免出现重复预订或超额预订。

2. 预订状态管理需求  
   系统应对预订全过程进行状态管理与跟踪，包括预订创建、确认、取消、入住、完成等环节。预订状态变化应能够形成日志记录，便于后续查询和追溯。系统还应保证预订状态与房间状态保持一致，例如预订创建后需要锁定房间，预订取消或完成后应释放房间资源。

3. 入住办理需求  
   当客户到店后，员工应能够根据预订编号或客户信息快速查询有效预订，并在核验无误后完成入住办理。系统应支持生成入住主记录和入住人明细记录，登记入住人员姓名、证件信息、联系电话等内容，并同步更新预订状态和房间状态，使预订信息能够顺利转化为实际入住业务。

4. 退房结算需求  
   客户退房时，系统应能够根据预订记录、入住记录和费用规则自动生成账单。账单内容应至少包括房费、服务费、优惠金额、应付金额、已付金额和退款金额等字段，并能够支持未支付、部分支付、已支付和已退款等多种账单状态。退房完成后，系统还应同步释放房间资源，并保留退房过程相关业务记录。

5. 支付与退款需求  
   系统应支持现金、银行卡、支付宝、微信等多种支付方式，并记录每一笔支付的支付流水号、支付金额、支付时间和支付状态。账单可以支持一次支付，也应支持部分支付和多次支付。对于发生退款的业务，系统应记录退款金额、退款原因、退款状态和对应支付记录，并自动维护账单退款金额和账单状态，保证支付数据与账单数据一致。

6. 客房与房态管理需求  
   系统应能够维护房型、客房目录、实体房间和价格策略等基础数据，并对房间状态进行统一控制。房间状态至少应能反映空闲、已预订、已入住、清洁中和维修中等业务状态。系统既要支持业务过程中的自动状态联动，也要支持员工或管理员根据实际经营情况进行房态干预和维护。

7. 评价与统计分析需求  
   系统应支持客户对酒店和客房进行评价，并能够记录评分、评价内容和回复信息。评价提交后，系统应自动更新酒店或客房的评价数量和平均评分。与此同时，系统还应基于预订、入住、支付、退款、评价和操作日志等数据，形成客户价值分析、月度营收统计、员工绩效统计和客房周转分析等结果，为酒店经营决策提供数据支持。

8. 权限控制与日志管理需求  
   系统应针对客户、员工和管理员设置不同的操作权限。客户主要使用前台服务，员工负责业务处理，管理员负责后台基础数据与权限维护。系统应保留关键业务操作日志，能够记录操作人、业务类型、操作内容和时间信息，以便进行过程监管、问题追溯和责任认定。

9. 数据一致性与可维护性需求  
   系统应保证预订、入住、账单、支付、退款、评价等核心业务数据之间的逻辑一致性。对于房态更新、账单金额汇总、评分统计等一致性要求较高的场景，应通过触发器、存储过程、事务控制和状态日志等方式进行约束和维护。同时，系统设计应具有良好的可扩展性，便于后续继续补充统计分析、经营决策和辅助管理等功能。

### 3.2 业务参与角色

系统主要包含三类角色：

- 管理员：负责酒店资料、房型、客房、员工、公告、系统配置和权限信息等后台基础数据维护，同时查看统计报表并进行综合管理。
- 员工：负责客户预订处理、入住办理、退房结算、房态维护和日常业务操作，是系统核心业务执行者。
- 客户：负责浏览酒店与客房信息、提交预订申请、查看订单、支付账单、收藏客房并发表评价。

| 退款金额 | `refund_amount` | decimal | 10,2 | 否 | 0.00 | 大于等于 0 | 表示账单或退款记录中的累计退款金额。 |
| 支付金额 | `pay_amount` | decimal | 10,2 | 否 | 无 | 大于 0 | 表示单笔支付记录对应的实际支付金额。 |
| 评分值 | `score` | tinyint | 3 | 否 | 5 | 1-5 分 | 表示用户对酒店或客房的评价分值，用于评分统计和满意度分析。 |
| 评价内容 | `content` | longtext | - | 否 | 无 | 文本型 | 表示用户提交的评价正文内容，是评价分析的重要依据。 |
| 回复内容 | `reply` | longtext | - | 是 | NULL | 文本型 | 表示管理员或员工对评价的回复内容。 |
| 收藏对象编号 | `refid` | bigint | 20 | 否 | 无 | 逻辑外键 | 表示被收藏、被评价的酒店或客房对象编号。 |
| 操作类型 | `action` | varchar | 100 | 否 | 无 | 业务非空 | 用于记录员工或系统执行的关键业务操作。 |
| 操作备注 | `remark` | varchar | 500 | 是 | NULL | 文本型 | 用于记录房态维护、支付退款、日志审计等业务的补充说明。 |

| 关系 | 说明 |

### 4.2 各模块功能概述

前台服务功能模块面向游客和用户开放，主要承担信息浏览、客房预订和互动反馈等功能。门户信息浏览用于集中展示酒店和客房相关信息，使用户能够快速了解酒店整体服务内容；客房预订服务用于支持房态检索、价格查询和在线提交预订申请，是前台业务的核心入口；用户互动反馈用于记录客户对酒店和客房的评价、收藏以及预订结果反馈，为后续统计分析和服务改进提供数据基础。

用户信息管理功能模块主要负责系统前台用户的身份建立、资料维护和个人业务查询。账户注册与认证用于完成游客注册、用户登录和身份信息核验，保证系统用户身份的真实性和可识别性；个人资料维护用于管理用户基本资料、联系方式和账号安全信息；个人业务查询用于支持用户查看个人预订状态、入住记录、账单信息和消费情况，使用户能够清晰掌握自己的业务办理过程和历史记录。

客房业务管理功能模块是酒店管理系统的基础业务模块，主要负责客房资源、价格策略和房态控制等内容。客房资源配置用于维护房型分类、客房目录、实体房间档案和房间编号信息，确保酒店客房资源能够被准确记录和调用；房价策略管理用于维护基础房价、动态价格策略和价格规则，为不同房型、不同时段和不同经营策略提供价格支持；房态控制管理用于统一维护房间状态以及清洁、维修等特殊状态，保证客房资源在预订、入住和退房各环节中的状态变化清晰可追踪。

预订与入住管理功能模块主要负责客户预订申请、入住接待和退房联动处理等业务。预订业务处理用于受理客户预订申请、审核预订信息、锁定预订房态以及处理取消订单，保证预订流程完整有序；入住接待管理用于完成预订记录核销、入住人员登记、入住安排办理和入住凭证生成，使预订信息能够顺利转化为实际入住业务；退房联动处理用于登记退房状态、自动释放房态、归档入住记录并衔接后续账单结算，是连接入住管理和财务结算的重要环节。

财务结算管理功能模块主要负责账单生成、支付处理和退款流转。账单生成管理用于根据房费、服务费用和优惠金额自动计算客户应付金额，并生成离店账单；支付业务管理用于录入支付记录、校验支付金额、更新支付状态并同步账单已付情况，保证账单状态能够随支付过程实时变化；退款业务管理用于登记退款申请、核销退款金额、同步退款状态并回写账单结果，确保支付与退款业务在数据层面保持一致。

后台基础管理功能模块主要负责系统后台的人员、内容和辅助业务管理。人员与权限管理用于维护员工档案、管理员账户和系统账户授权，保证系统不同角色拥有相应的操作范围；内容信息管理用于维护公告信息、酒店简介和系统配置，是前台展示内容和后台运行设置的主要数据来源；业务辅助管理用于记录操作日志、跟踪预订状态、保存房态变更记录和管理入住安排，为业务过程监管、问题追溯和系统维护提供支撑。

统计与分析功能模块主要负责对业务数据进行汇总、统计和分析，为系统管理和经营决策提供依据。评价统计分析用于汇总酒店评分、客房评分、好评度和评价数量，反映服务质量和客户满意度；客户价值分析用于统计客户预订次数、入住次数、消费金额和客户画像，帮助系统识别重要客户和消费特征；经营决策支持用于统计营收状况、客房周转情况、房态利用情况和员工业务考评结果，为酒店管理人员开展经营分析和管理决策提供数据参考。

## 5. 业务流程讲解参考

### 5.1 客房查询与预订流程

客户进入系统后，可以查看酒店简介和客房列表。系统从 `room_catalog` 查询客房目录，从 `room_instance` 查询具体可用房间，并结合 `price_policy` 计算当前价格。客户选择客房后提交预订，系统调用或模拟 `sp_create_booking_order` 完成预订创建。

该流程的数据变化如下：

1. 客户从前台页面提交房型、日期、数量等预订信息。
2. 系统查询 `customer` 确认客户身份。
3. 系统查询 `room_catalog` 获取房型、价格、酒店地址等信息。
4. 系统查询 `room_instance` 选择一个空闲房间。
5. 系统向 `booking_order` 写入预订记录。
6. 系统向 `booking_status_log` 写入预订状态日志。
7. 系统更新 `room_instance.status` 为 `reserved`。
8. 系统向 `room_status_log` 和 `operation_log` 写入日志。

PowerDesigner DFD 建议：

- 外部实体：客户
- 处理过程：查询可订房间、创建预订、锁定房间
- 数据存储：客户表、客房目录表、具体房间表、预订表、预订状态日志、房态日志
- 数据流：查询条件、可订房间列表、预订申请、预订结果

### 5.2 入住办理流程

客户到店后，员工根据预订编号或客户信息查询预订记录。如果预订有效，员工确认入住并登记入住人信息。系统生成 `checkin_record` 和 `checkin_guest`，并将房间状态改为入住中。

该流程的数据变化如下：

1. 员工输入预订编号或客户信息。
2. 系统查询 `booking_order` 获取预订记录。
3. 系统确认 `room_instance` 中对应房间可入住。
4. 系统向 `checkin_record` 写入入住主记录。
5. 系统向 `checkin_guest` 写入入住人明细。
6. 系统更新 `booking_order.booking_status` 为 `checked_in`。
7. 系统更新 `room_instance.status` 为 `occupied`。
8. 系统写入房态日志、预订状态日志和员工操作日志。

PowerDesigner DFD 建议：

- 外部实体：员工、客户
- 处理过程：查询预订、办理入住、登记入住人、更新房态
- 数据存储：预订表、入住记录表、入住人明细表、具体房间表、房态日志表
- 数据流：入住申请、预订信息、入住登记信息、入住结果

### 5.3 退房结算与支付流程

客户退房时，员工查询入住记录并生成账单。账单包括房费、服务费、优惠金额和应付金额。客户支付后，系统写入支付记录，并通过触发器自动更新账单状态。

该流程的数据变化如下：

1. 员工根据房号或客户信息查询 `checkin_record`。
2. 系统根据 `booking_order` 计算房费。
3. 员工输入服务费、优惠金额等信息。
4. 系统调用或模拟 `sp_generate_checkout_bill` 生成 `bill`。
5. 客户完成支付后，系统调用或模拟 `sp_pay_bill` 写入 `payment_record`。
6. 支付触发器 `trg_payment_after_insert_update_bill` 自动更新 `bill.paid_amount` 和 `bill.status`。
7. 若发生退款，写入 `refund_record`，退款触发器自动更新账单退款金额和状态。

PowerDesigner DFD 建议：

- 外部实体：客户、员工、支付平台
- 处理过程：生成账单、支付账单、退款处理、收入统计
- 数据存储：入住记录表、账单表、支付记录表、退款记录表
- 数据流：退房申请、账单信息、支付信息、退款信息、结算结果

### 5.4 评价与统计流程

客户入住或浏览后，可以对酒店和客房进行评价。评价写入数据库后，触发器自动统计评价数量和平均评分。同时，系统通过视图统计客户消费价值、员工办理业务数量和月度收入。

该流程的数据变化如下：

1. 客户提交酒店评价或客房评价。
2. 系统写入 `hotel_review` 或 `room_review`。
3. 评价触发器自动计算平均评分。
4. 系统更新 `hotel_profile.avg_score` 或 `room_catalog.avg_score`。
5. 管理员或员工通过统计视图查看经营情况。

PowerDesigner DFD 建议：

- 外部实体：客户、管理员
- 处理过程：提交评价、更新评分、生成统计报表
- 数据存储：酒店评价表、客房评价表、酒店表、客房目录表、统计视图
- 数据流：评价内容、评分数据、统计结果

### 7.3 CDM 到 LDM 转换说明

概念模型转逻辑模型时，需要把实体转换为关系模式，把实体属性转换为表字段，把实体之间的一对多关系转换为外键字段。例如：

- `room_instance.room_catalog_id` 引用 `room_catalog.id`
- `booking_order.room_instance_id` 引用 `room_instance.id`
- `checkin_record.booking_id` 引用 `booking_order.id`
- `checkin_record.room_id` 引用 `room_instance.id`
- `checkin_guest.checkin_id` 引用 `checkin_record.id`
- `bill.checkin_id` 引用 `checkin_record.id`
- `payment_record.bill_id` 引用 `bill.id`
- `refund_record.payment_id` 引用 `payment_record.id`

部分外键没有强制声明为数据库级 `FOREIGN KEY`，但在逻辑设计和物理模型图中应按照上述依赖关系表达。报告中可以说明：部分外键约束采用“逻辑外键 + 索引 + 存储过程/触发器维护”的方式实现。

## 9. 可直接写入报告的系统介绍示例

本系统为酒店管理数据库应用系统，面向酒店日常运营场景，围绕客户订房、员工办理入住、退房结算、房态维护和经营统计等业务建立统一的数据管理平台。

系统主要服务于管理员、员工和客户三类用户。管理员负责维护酒店基础资料、客房类型、客房信息、员工信息和公告内容；员工负责处理客户预订、入住登记、退房结算和房态调整；客户可以浏览酒店与客房信息，提交客房预订，查看订单状态，完成支付并发表评价。

在数据库设计方面，系统采用较为完整和规范的关系结构，包含具体房间、房态日志、预订状态日志、入住记录、入住人明细、账单、支付记录、退款记录、价格策略和操作日志等核心数据表，使系统能够覆盖客房资源、预订入住、财务结算和统计分析等主要业务内容。

系统通过视图实现复杂数据查询，例如房型库存统计、房态查询、预订明细、入住明细、账单支付汇总、月度收入统计、员工操作统计和客户价值分析。通过存储过程封装关键业务操作，例如查询可订房间、创建预订、办理入住、生成账单和支付账单。通过触发器维护关键数据一致性，例如预订后锁定房间、取消预订后释放房间、入住后更新房态、支付后更新账单状态、评价后更新平均评分等。

因此，本系统既能够完成简单的前后台交互，也能够体现数据库课程设计中对需求分析、概念结构设计、逻辑结构设计、物理结构设计、视图、存储过程和触发器的综合应用。
