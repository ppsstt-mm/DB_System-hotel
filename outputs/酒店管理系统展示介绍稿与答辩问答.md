# 酒店管理系统展示介绍稿与答辩问答

## 一、系统总体介绍

本系统是一个围绕酒店业务设计的数据库课程设计系统。它不是单纯做一个网页，而是把数据库中的表结构、视图、存储过程、触发器、约束和业务关系，通过前台客户页面和后台管理页面可视化地展示出来。

系统分为三个层次：

- 前台客户端：面向客户，支持浏览房型、查看酒店信息、提交预订、支付订单、查看订单状态和维护入住人信息。
- 后台管理端：面向管理员或员工，支持查看业务表、数据库视图、订单流转、房态、账单、支付、清扫、维修等数据。
- 数据库层：以 MySQL 为核心，保存业务数据，并通过视图、存储过程、触发器和约束体现数据库设计。

可以这样介绍本系统的定位：

> 我们的系统重点不是只做页面，而是把数据库设计转化成一个可以操作、可以展示、可以验证的可视化系统。前端页面负责展示和交互，后端负责接口和业务调用，数据库负责保存数据、组织关系、封装流程和保证一致性。

## 二、使用技术

### 1. 后端技术

后端使用 Spring Boot 作为主要框架，负责提供 HTTP 接口、处理登录权限、接收前端请求、调用数据库并返回 JSON 数据。

后端还使用了：

- Maven：负责 Java 项目的依赖管理和打包构建。
- MyBatis Plus：用于通用表的增删改查，减少重复代码。
- JDBC：用于复杂 SQL、视图查询和存储过程调用。
- MySQL Driver：用于连接 MySQL 数据库。

Spring Boot 与数据库的联系是：Spring Boot 通过数据库连接配置连接 MySQL，然后通过 MyBatis Plus 或 JDBC 执行 SQL、查询视图、调用存储过程、更新数据表。网页上的操作会先进入 Spring Boot 后端，再由后端访问数据库。

### 2. 前端技术

系统有两个主要前端：

- 前台客户页面：使用 HTML、CSS、JavaScript 实现，运行在 Spring Boot 静态资源目录中。
- 后台管理页面：使用 Vue2 + Element UI 实现，运行在 `8081` 端口，主要用于后台管理和数据库展示。

前端和后端通过 HTTP 接口通信。前端不直接连接数据库，而是向后端发送请求，后端再访问数据库。

例如客户点击“支付”：

1. 前台页面发送支付请求到 Spring Boot 后端。
2. 后端校验订单和用户权限。
3. 后端写入支付流水、更新账单状态、更新订单支付状态。
4. 数据库更新成功后返回结果。
5. 前端刷新页面，显示“已支付”“待入住”等状态。

### 3. 数据库技术

数据库使用 MySQL，主要体现了以下设计：

- 基础表：保存客户、员工、房型、实体房间、订单、入住记录、账单、支付记录、清扫任务、维修记录等业务数据。
- 视图：整合多张表，为后台页面提供综合查询结果。
- 存储过程：封装复杂业务流程，例如创建预订、办理入住、生成账单、账单收款、批量调整房态。
- 触发器和约束：保证部分状态联动和数据合法性。
- 索引和外键：提高查询效率，并表达实体之间的关系。

数据库在本系统中的作用不是简单保存数据，而是整个系统的核心。前端看到的订单状态、房间状态、账单状态、支付状态，本质上都来自数据库表和视图。

## 三、数据库和网页的关系

本系统可以理解为数据库设计的可视化体现。

数据库中的基础表对应后台的业务管理页面，例如：

- `booking_order` 对应后台“预订订单”页面。
- `room_instance` 对应后台“实体客房/房态”页面。
- `booking_room` 对应“订单房间明细”页面。
- `booking_guest` 对应“预订入住人”页面。
- `checkin_record` 对应“入住记录”页面。
- `bill` 对应“账单台账”页面。
- `payment_record` 对应“支付记录”页面。
- `housekeeping_task` 对应“清扫任务”页面。
- `maintenance_record` 对应“维修记录”页面。

网页上修改数据后，数据库可以看到对应变化。比如：

- 前台创建订单后，数据库会新增订单、订单房间明细、入住人和账单记录。
- 前台支付订单后，数据库会新增支付流水，并更新账单和订单支付状态。
- 后台调整房态后，数据库中的实体房间状态会变化。
- 后台创建清扫任务后，数据库中的清扫任务表会新增记录。

反过来，如果直接在数据库里修改数据，刷新网页后也能看到最新结果。不过正式系统中不建议直接手动改库，因为这样可能绕过业务规则。

当前系统的重点是在线展示和筛选，并没有把“下载文件”作为核心功能。答辩时可以这样说：

> 当前系统主要实现了数据库视图的在线可视化展示、分页和筛选。下载导出不是本次课程设计的重点，但后续可以基于已有接口增加 Excel 或 CSV 导出功能。

## 五、存储过程在页面中的体现

存储过程集中体现在后台“流程中心”页面：

```text
http://localhost:8081/#/procedure_center
```

流程中心把数据库中的存储过程封装成表单和按钮。用户填写参数并点击按钮，前端调用 Spring Boot 接口，Spring Boot 再调用 MySQL 存储过程。

### 1. 房源与预订

页面标签：

```text
房源与预订
```

体现的存储过程：

- `sp_search_available_rooms`：可用房源检索。
- `sp_create_booking_order`：存储过程创建预订。

可以展示：输入房型、入住日期、离店日期、价格范围，查询可用房源；输入客户 ID、客房目录 ID、数量、预订日期，创建预订订单。

### 2. 入住与结算

页面标签：

```text
入住与结算
```

体现的存储过程：

- `sp_confirm_checkin`：办理入住。
- `sp_generate_checkout_bill`：生成退房账单。
- `sp_pay_bill`：账单收款。

可以展示：办理入住后，入住记录、房间状态和订单状态发生变化；退房时生成账单；收款后账单和支付流水同步更新。

### 3. 房态与服务

页面标签：

```text
房态与服务
```

体现的存储过程：

- `sp_batch_update_room_status`：批量房态调整。
- `sp_create_housekeeping_task`：创建清扫任务。
- `sp_report_room_maintenance`：上报维修。

可以展示：房间从可预订到已预订、入住中、清扫中、维修中等状态变化。

### 4. 统计分析

页面标签：

```text
统计分析
```

体现的存储过程：

- `sp_customer_consumption_report`：客户消费统计。
- `sp_employee_performance_report`：员工绩效统计。
- `sp_generate_employee_assessment`：员工考核生成。

可以展示：数据库不仅支持业务操作，也支持统计和考核。

## 六、主要业务线

答辩时推荐围绕“预订到入住再到账单结算”这条主线讲，因为它最能体现数据库设计。

### 业务线 1：客户浏览房型并创建预订

前台页面：

```text
http://localhost:8080/springbootb1g8z/front/pages/room_catalog/list.html
```

客户浏览房型，进入详情页，选择预计入住和离店日期，提交预订。

数据库变化：

- `booking_order` 新增订单主表记录。
- `booking_room` 新增订单和实体房间的对应关系。
- `booking_guest` 新增入住人记录。
- `bill` 生成初始账单。
- `room_instance` 对应房间状态变为已预订。

可以强调：这里体现的是订单、房型、实体房间、客户、入住人之间的关系。

### 业务线 2：订单支付

前台页面：

```text
http://localhost:8080/springbootb1g8z/front/pages/booking_order/list.html
```

客户进入订单列表或订单详情，完成支付。

数据库变化：

- `payment_record` 新增支付流水。
- `bill` 更新实收金额和账单状态。
- `booking_order` 更新支付状态。
- `room_instance` 更新时间刷新，后台房态页可以按更新时间看到最近变化的房间。

可以强调：支付不是页面文字变化，而是账单、流水、订单、房态多个数据对象同步变化。

### 业务线 3：后台验证订单和房态

后台页面：

```text
http://localhost:8081/#/booking_order
http://localhost:8081/#/room_instance
http://localhost:8081/#/booking_room
http://localhost:8081/#/v_booking_detail
```

可以展示订单主表、实体房间状态、订单房间明细和预订明细视图。

这里适合说明：前台创建和支付的结果，可以在后台数据库可视化页面中验证。

### 业务线 4：办理入住和退房结算

后台页面：

```text
http://localhost:8081/#/procedure_center
http://localhost:8081/#/checkin_record
http://localhost:8081/#/v_checkin_guest_detail
http://localhost:8081/#/bill
http://localhost:8081/#/v_bill_payment_summary
```

通过流程中心办理入住，再生成退房账单和登记收款。

数据库变化：

- `checkin_record` 新增入住记录。
- `checkin_guest` 记录实际入住人。
- `booking_order` 订单状态变化。
- `room_instance` 房态变化。
- `bill` 和 `payment_record` 记录结算和支付。

### 业务线 5：房态服务闭环

后台页面：

```text
http://localhost:8081/#/housekeeping_task
http://localhost:8081/#/maintenance_record
http://localhost:8081/#/room_status_log
http://localhost:8081/#/v_housekeeping_task_detail
```

房间退房后可能进入清扫，发现问题后可能进入维修，维修完成后再回到可预订状态。

可以强调：房态不是一个孤立字段，而是由预订、入住、清扫、维修多个业务共同驱动。

## 七、展示时可以采用的讲解顺序

建议现场展示顺序如下：

1. 打开前台房型列表，说明客户可以浏览房型并发起预订。
2. 打开前台订单列表，说明订单支付状态和订单追踪来自数据库。
3. 打开后台 `booking_order`，展示订单主表。
4. 打开后台 `room_instance`，展示实体房间和房态排序。
5. 打开后台 `v_booking_detail`，展示数据库视图把客户、订单、房间整合到一起。
6. 打开 `procedure_center`，展示存储过程如何变成可操作按钮。
7. 打开 `v_bill_payment_summary`，展示账单、支付和退款汇总。
8. 打开清扫/维修相关页面，说明业务闭环。

一句总结：

> 前台负责把业务流程展示给用户，后台负责把数据库结构和业务数据展示给管理员，数据库负责保证数据关系、状态流转和统计结果。

## 八、基础答辩问答

### Q1：你们数据库设计体现在哪里？

A：主要体现在三层：第一是基础表，保存客户、房间、订单、入住、账单、支付、清扫、维修等业务实体；第二是视图，把多表关联和统计结果展示出来；第三是存储过程，把创建预订、办理入住、账单结算等复杂流程封装在数据库中。

### Q2：为什么要使用视图？

A：因为很多管理页面不是单表能完成的。例如预订明细要关联客户、订单、房间和房型；账单汇总要关联账单、支付流水和退款记录。视图可以统一查询口径，减少前端和后端重复拼接数据。

### Q3：为什么要使用存储过程？

A：存储过程适合处理跨表、强一致的业务流程。例如办理入住不是只插入一条入住记录，还要更新订单状态、房态、入住人、日志等数据。把这些逻辑封装在数据库中，更能体现数据库课程设计的完整性。

### Q4：系统最能体现数据库设计的一条业务是什么？

A：预订到入住再到账单结算。这条线贯穿 `booking_order`、`booking_room`、`room_instance`、`booking_guest`、`checkin_record`、`bill`、`payment_record` 等表，同时还能通过视图和日志验证业务结果。

### Q5：数据库和前端页面是什么关系？

A：前端页面展示的是数据库中的数据，用户在页面上的操作会通过后端写入数据库。比如订单支付后，页面显示“已支付”，实际是数据库中的支付流水、账单状态和订单支付状态发生了变化。

## 九、技术答辩问答

### Q1：这个数据库是部署在哪里的？

A：目前数据库部署在本机 MySQL 上，也就是本地数据库。Spring Boot 后端通过配置文件连接本机的 `hotel_management` 数据库。演示时前端、后端和数据库都运行在同一台电脑上。

### Q2：其他人可以使用这个系统吗？

A：可以，但要看网络环境和访问地址。如果其他设备和本机在同一个 WiFi 下，并且本机防火墙和端口允许访问，就可以通过本机局域网 IP 访问系统。例如：

```text
http://本机IP:8080/springbootb1g8z/
http://本机IP:8081/#/
```

目前系统更接近局域网部署，不是公网服务器部署。

### Q3：如果要让外网访问，需要做什么？

A：需要把前端、后端和数据库部署到云服务器，或者使用内网穿透、端口映射等方式。同时需要配置公网 IP、域名、防火墙、安全组。正式上线时通常不会把 MySQL 端口直接暴露给公网，而是只允许后端服务访问数据库。

### Q4：除了 Spring Boot，还用了哪些技术？

A：后端还使用了 Maven、MyBatis Plus、JDBC、MySQL Driver；后台前端使用 Vue2、Element UI、Axios；前台客户页面使用 HTML、CSS、JavaScript。Maven 负责依赖管理和打包，MyBatis Plus/JDBC 负责访问数据库。

### Q5：Maven 的作用是什么？

A：Maven 负责管理 Java 依赖和项目构建。Spring Boot、MySQL 驱动、MyBatis Plus 等依赖都通过 Maven 配置引入。运行 `mvn package` 可以把后端项目打包成可运行的 jar。

### Q6：Spring Boot 和数据库的联系是什么？

A：Spring Boot 是后端服务框架，数据库是数据存储和业务规则承载层。Spring Boot 通过数据库连接配置和 MySQL 驱动连接数据库，再通过 MyBatis Plus 或 JDBC 执行 SQL、查询视图、调用存储过程。

### Q7：前端和后端是什么关系？

A：前端负责展示页面和收集操作，后端负责业务处理和数据库访问。前端不会直接连接数据库，而是通过 HTTP 请求调用 Spring Boot 后端接口。

### Q8：为什么不让前端直接连接数据库？

A：因为不安全。前端代码会暴露给用户，如果直接连接数据库，数据库账号、密码和 SQL 权限都会暴露。正确方式是前端请求后端接口，由后端统一鉴权、校验和访问数据库。

### Q9：网页上直接修改，数据库里能看到吗？

A：可以。比如后台修改房间状态，数据库 `room_instance` 表会变化；新增清扫任务，`housekeeping_task` 表会新增记录；前台支付订单，数据库中的 `payment_record`、`bill`、`booking_order` 会变化。

### Q10：数据库里直接改数据，网页会同步吗？

A：会，但需要刷新页面或重新查询。网页上的数据来自后端查询数据库，所以数据库修改后，前端重新加载列表就能看到最新结果。不过正式系统不建议绕过页面直接改库。

### Q11：这个系统是前后端分离吗？

A：整体是前后端分离思路。后台 Vue 运行在 `8081`，通过接口访问 `8080` 的 Spring Boot 后端；前台客户页面由 Spring Boot 静态资源提供，也通过后端接口访问数据。

### Q12：为什么后台是 8081，前台是 8080？

A：`8080` 是 Spring Boot 后端服务端口，同时提供前台静态页面和接口；`8081` 是 Vue 后台管理端的开发服务器端口。后台页面本身运行在 `8081`，但数据请求仍然会发到 `8080`。

### Q13：MyBatis Plus 和 JDBC 有什么区别？

A：MyBatis Plus 适合通用表的增删改查，可以减少重复代码；JDBC 更适合复杂 SQL、存储过程调用和特殊查询。本系统两者都使用了。

### Q14：数据库视图和普通表有什么区别？

A：普通表保存真实业务数据，视图不直接保存数据，而是基于表查询出来的结果。比如 `v_booking_detail` 把订单、客户、房间、房型整合成一张展示视图。

### Q15：存储过程和后端接口有什么区别？

A：后端接口是 Java 层对外提供的入口，存储过程是数据库内部封装的业务逻辑。比如后台点击“办理入住”，前端调用 Spring Boot 接口，Spring Boot 再调用 MySQL 的 `sp_confirm_checkin`。

### Q16：系统支持多用户吗？

A：支持基本多用户。系统有客户、管理员、员工等角色数据，客户可以前台登录查看自己的订单，后台管理端可以查看和维护业务数据。课程设计层面已经体现了角色和数据隔离。

### Q17：多人同时使用会不会影响数据库？

A：多人操作会写入同一个 MySQL 数据库。数据库本身支持并发访问；关键业务可以通过事务、约束、存储过程和状态字段来降低数据不一致风险。

### Q18：这个系统的数据是实时的吗？

A：是请求级实时。页面每次刷新或点击查询都会请求后端接口，后端查询数据库返回最新数据。它不是 WebSocket 自动推送，但对管理系统来说，刷新后看到的就是当前数据库状态。

### Q19：怎么证明网页操作真的落到了数据库？

A：可以现场演示。比如前台创建订单或支付订单后，到后台 `预订订单`、`实体客房`、`账单支付汇总`、`支付记录` 页面查看变化；也可以直接用 MySQL 查询对应表，看到记录同步变化。

### Q20：数据库在系统里的最大作用是什么？

A：数据库是系统的核心，不只是保存数据。它定义了业务对象之间的关系，保证状态联动，提供视图统计，封装存储过程流程，并支撑前台和后台所有页面的数据展示。

## 十、答辩结尾总结

可以用下面这段作为结尾：

> 本系统以酒店管理业务为背景，将客户、客房、预订、入住、账单、支付、清扫和维修等对象进行了数据库建模。前台页面体现客户业务流程，后台页面体现数据库表、视图和存储过程的可视化展示。系统通过 Spring Boot 连接数据库，通过 Vue 和 HTML 页面展示数据，最终实现了一个以数据库设计为核心、以网页系统为表现形式的综合课程设计。
